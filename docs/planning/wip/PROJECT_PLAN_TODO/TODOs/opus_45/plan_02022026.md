# Focus Guard MVP Roadmap - February 2026

## Executive Summary

This plan outlines the three critical components needed to reach a deployable MVP:

1. **Configuration UI** - A simple interface for users to configure rules, budgets, and settings
2. **Windows Executable Packaging** - Standalone .exe that runs without Python installation
3. **Browser Extension Store Submission** - Chrome Web Store and Edge Add-ons readiness

**Target Completion**: 2-3 weeks

---

## Current State Assessment

### What's Working ✅
- Activity monitoring with idle detection
- Classification system (YouTube, Reddit, Twitter, Google + generic URL)
- Content-type aware budgets (EDUCATION: 60min, ENTERTAINMENT: 10min, etc.)
- Override system with classification integration
- Browser extension (MV3) with blocking, popup, and blocked page
- Tab server with HTTP API for extension communication
- Email reporting system
- Basic PyInstaller build scripts

### What's Missing ❌
- No user-facing configuration UI
- .exe packaging not fully tested/documented for end-users
- Browser extension not submitted to stores (missing icons, privacy policy, store listings)
- No installer that bundles everything together
- Log cleanup/storage management
- API key handling for LLM classification

## Phase 2: Windows .exe Packaging
**Effort: 2-3 days | Priority: HIGH**

### 2.1 Architecture Decision

**Decision: Single Executable**

One `FocusGuard.exe` that includes everything:
- Tab server (HTTP API for browser extension)
- Activity monitoring service
- Streamlit configuration UI
- System tray icon for status/control
- Log cleanup scheduler

**Characteristics:**
- File size: ~100-150MB (includes Python runtime, Streamlit, dependencies)
- Self-contained: No Python installation required
- Must run as Administrator for full functionality
- Starts tab server on configurable port (default: 58392)

**Future Enhancement (Post-MVP):**
- Inno Setup installer for easier deployment
- Windows Service option for auto-start
- Separate tray app for lighter resource usage

### 2.8 Classification Proxy Service

**Problem**: Cannot distribute application with developer's OpenAI API key, but requiring users to set up their own API account adds friction and reduces adoption.

**Solution**: Host a lightweight classification proxy service that the app calls.

```
┌─────────────┐     ┌──────────────────────┐     ┌─────────────┐
│ Focus Guard │────▶│ api.focusguard.com   │────▶│ OpenAI API  │
│   (Client)  │     │ (Your Proxy Server)  │     │ (or other)  │
└─────────────┘     └──────────────────────┘     └─────────────┘
                              │
                              ▼
                    ┌──────────────────────┐
                    │ - Advanced rules     │
                    │ - Caching layer      │
                    │ - Rate limiting      │
                    │ - Analytics          │
                    └──────────────────────┘
```

**Benefits:**
- Zero setup for users (no API key needed)
- You control the model provider (can switch OpenAI → Anthropic → local)
- Advanced server-side rules before hitting LLM
- Caching reduces costs dramatically
- Collect anonymized data to improve classification rules
- Rate limiting prevents abuse

**Architecture:**

1. **Client-side flow:**
```python
def classify_url(url: str, domain: str) -> ClassificationResult:
    # Step 1: Check local rules (YouTube, Reddit, Twitter, etc.)
    result = local_classifier.classify(url, domain)
    if result.confidence > 0.8:
        return result
    
    # Step 2: Check local cache
    cached = cache.get(domain)
    if cached:
        return cached
    
    # Step 3: Call proxy service
    result = proxy_service.classify(url, domain)
    cache.set(domain, result, ttl=7_days)
    return result
```

2. **Server-side (Proxy Service):**
```python
# Serverless function (Cloudflare Worker / Vercel Edge / AWS Lambda)
def handle_classification(request):
    url = request.json["url"]
    domain = request.json["domain"]
    
    # Step 1: Server-side rules (more comprehensive than client)
    result = server_rules.classify(url, domain)
    if result:
        return result
    
    # Step 2: Check server cache (Redis/KV store)
    cached = kv_store.get(f"classification:{domain}")
    if cached:
        return cached
    
    # Step 3: Call LLM API
    result = openai_classify(url, domain)
    kv_store.set(f"classification:{domain}", result, ttl=30_days)
    
    # Step 4: Log for analytics (anonymized)
    log_classification(domain, result)  # No user info
    
    return result
```

**Hosting Options (Low Maintenance):**

| Option | Cost | Maintenance | Pros | Cons |
|--------|------|-------------|------|------|
| **Cloudflare Workers** | Free tier: 100K req/day | Very Low | Edge caching, KV store included, auto-scaling | 10ms CPU limit per request |
| **Vercel Edge Functions** | Free tier: 100K req/month | Very Low | Easy deploy from Git, good DX | Less generous free tier |
| **AWS Lambda + API Gateway** | ~$0.20/million requests | Low | Most flexible, can use RDS | More setup required |
| **Railway / Render** | $5-10/month | Low | Simple deployment, always-on | Fixed cost even if unused |

**Recommended: Cloudflare Workers + KV**
- Free tier handles thousands of users
- Built-in caching (KV store)
- Global edge deployment (fast everywhere)
- Zero server maintenance
- Easy to deploy and update

**Cost Estimation:**

```
Assumptions:
- 1,000 active users
- 50 unique domains/user/day = 50,000 domain lookups/day
- 90% hit local rules or cache = 5,000 proxy calls/day
- 80% hit server cache = 1,000 LLM calls/day
- GPT-4o-mini: ~$0.00015/classification

Daily LLM cost: 1,000 × $0.00015 = $0.15/day
Monthly cost: ~$5/month for 1,000 users

With better caching (domain-level, 30-day TTL):
- Most domains classified once ever
- Steady-state: <$1/month
```

### 3.9 Extension Update Policy

**What Triggers Re-Review:**
- Adding new permissions in manifest.json
- Changing `host_permissions`
- Major functionality changes
- Changes to data collection practices

**What Does NOT Trigger Re-Review:**
- Bug fixes in JavaScript code
- UI improvements (CSS, HTML)
- Performance optimizations
- Adding new features that use existing permissions
- Updating version number

**Update Process:**
1. Make code changes
2. Increment version in `manifest.json` (e.g., 1.0.0 → 1.0.1)
3. Create ZIP of extension folder
4. Upload to Chrome Web Store / Edge Add-ons dashboard
5. Submit for review (usually faster than initial review: hours to 1 day)

**Best Practices:**
- Get all permissions right in the initial submission
- Use broad permissions initially if you might need them later
- Document why each permission is needed
- Keep a changelog for store updates
